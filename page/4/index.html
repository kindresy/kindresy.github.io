中断上下半部

上半部和下半部的定义区别

上半部是处理紧急的任务，比如硬件相关的操作，需要快速执行然后释放中断。这部分通常是在中断上下文中执行的，不能休眠或者进行可能导致阻塞的操作。那下半部呢，就是处理那些不那么紧急的、耗时较长的任务，比如数据处理或者触发软中断。这部分可以延迟执行，允许重新开启中断，从而提升系统的实时性。

实现方法

Linux中有几种常见的机制，比如软中断（softirq）、tasklet和工作队列（workqueue）。软中断是静态分配的，运行在中断上下文中，不能休眠，而tasklet是基于软中断实现的，但同一时间只能在一个CPU上运行，适合处理可延迟的任务。工作队列则是在进程上下文中执行，可以休眠，适合需要长时间处理的任务。

线程化中断（threaded IRQ）这也是现代内核中的一种方式，把下半部放到内核线程里处理，这样可以更灵活地管理优先级和调度。还有小任务（tasklet）和定时器（timer）也可能被用来实现下半部

Linux内核提供了多种下半部实现机制，适用于不同场景：

  1. 软中断（Softirq）
    特点：
    静态注册：编译时通过open_softirq()定义，如网络子系统（NET_RX_SOFTIRQ）、定时器（TIMER_SOFTIRQ）。
    无锁并发：同一软中断可在不同CPU上并行执行，需开发者自行处理竞态条件。
    执行时机：中断返回前（irq_exit()）或内核线程ksoftirqd中触发。
    适用场景：高频、低延迟任务（如网卡收包）。

  2. Tasklet
    特点：
    动态注册：通过tasklet_init()动态创建，基于软中断（TASKLET_SOFTIRQ/HI_SOFTIRQ）实现。
    串行执行：同一Tasklet在多个CPU上串行执行，天然避免并发问题。
    优先级区分：高优先级Tasklet（HI_SOFTIRQ）优先执行。
    适用场景：中低频任务（如USB设备中断处理）。

  3. 工作队列（Workqueue）
    特点：
    进程上下文执行：通过内核线程（如kworker）调度，允许休眠和阻塞操作。
    灵活性：支持延迟任务（delayed_work）、绑定特定CPU（queue_work_on()）。
    资源隔离：可创建专用工作队列（如alloc_workqueue()）避免相互影响。
    适用场景：复杂、长耗时任务（如文件系统操作）。

  4. 线程化中断（Threaded IRQ）
    特点：
    中断处理线程化：通过request_threaded_irq()注册，上半部仅确认中断，下半部由独立线程执行。
    实时性优化：可通过调整线程优先级（如SCHED_FIFO）提升响应速度。
    适用场景：需要复杂处理或与用户空间交互的中断（如GPIO中断）。

一、命令说明

mmc_go_idle 
发送CMD0指令，GO_IDLE_STATE 
使mmc card进入idle state。 
虽然进入到了Idle State，但是上电复位过程并不一定完成了，这主要靠读取OCR的busy位来判断，而流程归结为下一步。

mmc_send_op_cond 
发送CMD1指令，SEND_OP_COND 
这里会设置card的工作电压寄存器OCR，并且通过busy位（bit31）来判断card的上电复位过程是否完成，如果没有完成的话需要重复发送。 
完成之后，mmc card进入ready state。

mmc_all_send_cid 
这里会发送CMD2指令，ALL_SEND_CID 
广播指令，使card回复对应的CID寄存器的值。在这里就相应获得了CID寄存器的值了，存储在cid中。 
完成之后，MMC card会进入Identification State。

mmc_set_relative_addr 
发送CMD3指令，SET_RELATIVE_ADDR 
设置该mmc card的关联地址为card->rca，也就是0x0001 
完成之后，该MMC card进入standby模式。

mmc_send_csd 
发送CMD9指令，MMC_SEND_CSD 
要求mmc card发送csd寄存器，存储到card->raw_csd中，也就是原始的csd寄存器的值。 
此时mmc card还是处于standby state

mmc_select_card & mmc_deselect_cards 
发送CMD7指令，SELECT/DESELECT CARD 
选择或者断开指定的card 
这时卡进入transfer state。后续可以通过各种指令进入到receive-data state或者sending-data state依次来进行数据的传输

mmc_get_ext_csd 
发送CMD8指令，SEND_EXT_CSD 
这里要求处于transfer state的card发送ext_csd寄存器，这里获取之后存放在ext_csd寄存器中 
这里会使card进入sending-data state，完成之后又退出到transfer state。

mmc_switch 
发送CMD6命令，MMC_SWITCH 
用于设置ext_csd寄存器的某些bit

mmc_send_status 
发送CMD13命令，MMC_SEND_STATUS 
要求card发送自己当前的状态寄存器

mmc_send_cid 
发送CMD10命令，MMC_SEND_CID 
要求mmc card回复cid寄存器

mmc_card_sleepawake 
发送CMD5命令，MMC_SLEEP_AWAKE 
使card进入或者退出sleep state，由参数决定。

HSEM的驱动设计-接口设计-实现

1. HSEM硬件机制

硬件信号量模块(HSEM)。通过读取一个该模块的寄存器地址来申请信号量资源，地址计算公式为:
SEM_INDIRECTm ADDR = offset + 0x200 * n + 0x4 * m。
offset为一个寄存器固定地址偏移

在A核上，HSEM模块硬件上的拓扑结构为: HSEM接到GIC控制器，而GIC接到CPU Cluster。A核上，HSEM在GIC中一共有32个bit位的中断。

上述SEM_INDIRECTm ADDR计算公式的n，对应的是输出32bit中断的序号(Peri下16bit，AO下16bit)。m为信号量的id，系统中一共有128个信号量资源，即 0 < m < 128.

以Peri子系统下为例：16bit状态位也就对应着前面SEM_INDIRECTm计算公式中的n，而公式中的m则为可以使用的信号量的个数，一共有128个。

当软件上某个任务A以特定的n和m读取到这个地址时，意味着着gic的某一个中断位n去申请信号量m。如果这个信号量空闲，则会读取到1，并且获取到信号量。另一个任务B可能在A持有期间，也去申请同一个信号量m(此时m与上面的m相同，n不必相同，但要满足n<16)，这时它会读取到0，信号量已经被占用，此时硬件可以帮助任务B以中断n的身份去等待信号量的释放（硬件上有一个等待队列），任务B可以去做其他事务。
任务B可以提前用中断号n来向linux注册一个中断处理函数。当任务A释放信号量时，任务A要向SEM_INDIRECTm寄存器写1来做释放。一旦任务A释放信号量m，任务B会立即出发中断n。

2. 驱动方案设计

2.1 功能目标

HSEM应该分为两大部件：

  1. 资源管理模块
  2. 信号量控制模块

驱动要实现对128个信号量资源和中断bit位的管理，并且向其他客户驱动模块，提供申请信号量，释放信号量的接口，在申请信号量时，我们应该将中断n和信号量m标记为已经被占用，并且给予客户模块提供它的处理函数的机会，一旦客户模块无法申请到信号量，它可以做别的任务，直到信号量被释放，自动回调到它提供的处理函数。

最终实现的模块功能：

  • 实现多核环境下基于硬件的信号量同步机制
  • 支持 16 个中断通道 (n) × 128 个信号量 (m) 的管理
  • 提供阻塞/非阻塞式信号量申请接口
  • 通过中断通知机制实现异步信号量释放监听

Linux Kernel Driver Learning outline

第一章
字符驱动是内核中最简单一类驱动框架，通常会编译为模块的形式加载到内核，从字符驱动开始，了解内核驱动开发的编译加载流程，了解内核模块的加载机制，快速对内核驱动开发形成一些入门级的认识

第二章
设备驱动的软件架构思想。linux内核的移植性做的非常优秀，现代内核采用设备树+设备驱动的方法来实现驱动的高移植性和可裁剪性，这一章会学习设备驱动的软件架构思想，以及设备树机制，相关API的使用

第三章
Platform 总线是 Linux 设备驱动模型中用于管理片上系统（SoC）和非标准外设的核心机制，通过统一抽象实现硬件与驱动的解耦。常见的内核驱动都是以platform的形式管理

第四章
Linux 中断子系统是硬件与软件交互的核心枢纽，负责高效响应外设事件并协调CPU资源。Linux中断子系统一章中会我们会学习linux内核中断处理流程，中断上下半部，线程化中断（Threaded IRQ）和中断亲和性与负载均衡，会讲解GIC控制器模块以及其驱动，最终实现一个级联中断控制器。

第五章
同步与互斥。Linux 内核的同步与互斥机制是保障多核、多线程、中断上下文等复杂场景下数据一致性与执行有序性的核心基础设施。在这一章中，我们会学习同步与互斥的基本原理。了解为什么在内核中需要这些机制，在多核并发和中断处理过程中的并发问题案例。也会介绍内核提供的互斥机制如原子操作、自旋锁、信号量、互斥锁、RCU等。

第六章
Linux内存管理机制的认识。Linux 内存管理是操作系统核心功能之一，其设计目标在于高效利用物理资源、保障多任务隔离性、适配异构硬。在这一章中，我们会学习虚拟内存系统和物理内存管理。了解常见的内存分配算法以支持后续模块开发。

第七章
内核常见调试方法与异常定位手段。介绍异常调用栈，中断现场信息，addr2line，k-gdb，ftrace，strace等工具的用。

第八章
块设备驱动。块设备驱动是 Linux 存储生态的核心，负责管理硬盘、SSD、NVMe 等以固定大小数据块为访问单元的存储设备。在这一阶段我们会学习文件系统如UFS等，并学习mmc子系统，对代码的核心流程与请求链路进行跟踪，了解SDIO协议，移植sdhc驱动，并实现sdhc的tuning流程，支持HS400增强模式。最终还会对性能瓶颈做一下摸索，了解mmc_test的基准测试方法。

第九章
我们会学习自己写一个内存管理驱动模块的实现，实现用户态对内核态内存的管理。
